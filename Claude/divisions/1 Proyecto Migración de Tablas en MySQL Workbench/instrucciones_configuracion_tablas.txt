INTERMEDIARY - CONVENCION DE NOMBRES DE TABLAS
================================================
Empresa: IDG
Base de datos: intermediary_db
Motor: InnoDB desde MySQL 8.0.43
GUI: Workbench

== ESTRUCTURA DEL NOMBRE ==
t[###][tipo]_[nombre_tabla]

Donde:
- t        : prefijo fijo, indica que es una tabla
- ###      : numero de 3 dígitos con ceros a la izquierda (001, 012, 130)
- [tipo]   : letra que identifica el modulo al que pertenece la tabla
- _        : separador obligatorio
- [nombre] : nombre descriptivo en INGLÉS, palabras separadas por _

== RANGOS Y TIPOS ==
Compañía IDG   | letra: c | rango: 001 - 099 | ejemplo: t003c_roles
Transversal    | letra: t | rango: 101 - 199 | ejemplo: t130t_currencies
Empresa        | letra: e | rango: 203 - 497 | ejemplo: t212e_companies
Módulos        | letra: m | rango: 500 - 899 | ejemplo: t512m_product_groups
Temporales     | letra: p | rango: 900 - 999 | ejemplo: t912p_load_data


== REGLAS ==
- Todos los nombres de columnas en INGLÉS
- Los nombres después del _ van siempre en INGLÉS
- Palabras compuestas se separan con _ dentro del nombre
- Los rangos dan espacio suficiente para crecer sin reorganizar
- Nunca usar tildes, espacios ni caracteres especiales
- Siempre respetar el rango asignado a cada tipo de modulo
- No incluir bloque de encabezado con datos del script
- No incluir bloque de notas de revision
- No incluir comentarios de sección: PRIMARY KEY, UNIQUE KEYS, CHECK CONSTRAINTS
- No incluir comentario de fin de script
- Solo incluir comentarios de tipo -- ADVERTENCIA: cuando aplique

== COLUMNAS DE LAS TABLAS ==
- Todas tablas tiene un campo ID ( id int NOT NULL AUTO_INCREMENT COMMENT 'Consecutivo único del registro' ) y es un PRIMARY KEY (id)
- AUTO_INCREMENT: Si la tabla viene con INSERT de datos, conservar el valor de AUTO_INCREMENT original del DDL fuente, si la tabla no trae INSERT de datos, ajustar AUTO_INCREMENT=1 según la convención
- Todas tablas deben tener estos campos:
  `created_at`     datetime(6) NOT NULL COMMENT 'Fecha y hora de creación del registro',
  `updated_at`     datetime(6) NOT NULL COMMENT 'Fecha y hora de última actualización del registro',
  `created_by`     int         NOT NULL COMMENT 'Usuario que creó el registro',
  `updated_by`     int         NOT NULL COMMENT 'Usuario que realizó la última actualización del registro',
- Todos los nombres de columnas en INGLÉS
- Separador de palabras con _ (snake_case)
- Ejemplo: first_name, last_name, identification_number
- Clave primaria siempre: id (INT AUTO_INCREMENT PRIMARY KEY)
- Fechas de auditoria: created_at, updated_at
- Usuarios de auditoria: created_by, updated_by
- Estado lógico: status 1=activo, 0=inactivo
- Solo se debe usar campos de tipo varchar(##), int, datetime(6)  y longblob para los datos en base 64
- Todas las columnas son NOT NULL 
- Todas las columnas llevan el COMMENT en español y se debe respetar las tildes.
- Todas las tablas debe tener una columna de nombre id int NOT NULL AUTO_INCREMENT COMMENT 'Consecutivo único del registro'

== CONVENCION DE CONSTRAINTS ==

-- PREFIJOS --
UK_   Unique Key
FK_   Foreign Key
CHK_  Check Constraint
IDX_  Index

-- REGLAS PARA CHK --
NE    Not Empty - campo no puede ser vacio   ''
GT0   Greater Than Zero - campo debe ser mayor a cero   > 0
LEN   Length - validacion de longitud del campo   LENGTH()

prefijo_tabla:
t[###][tipo]_
Donde:
- t        : prefijo fijo, indica que es una tabla
- ###      : numero de 3 dígitos con ceros a la izquierda (001, 012, 130)
- [tipo]   : letra que identifica el modulo al que pertenece la tabla
- _        : separador obligatorio


-- FORMATO GENERAL --
[PREFIJO]_[REGLA]_[prefijo_tabla]_[columna]

-- INDEX --
formato : IDX_t[###][tipo]_[nombre_campo]
ejemplo : INDEX `IDX_t125t_code` (`code`)

-- PRIMARY KEY --
Todas las tablas DEBEN tener PRIMARY KEY en el campo id
formato : PRIMARY KEY (`[nombre_campo]`)
ejemplo : PRIMARY KEY (`id`)

-- FOREIGN KEY --
formato : FK_t[###][tipo]_[nombre_campo]_t[###][tipo]
ejemplo : CONSTRAINT `FK_t212e_id_idioma_t125t` FOREIGN KEY (`id_idioma`) REFERENCES `t125t_idioma` (`id`)

-- CHECK CONSTRAINT --
formato : CHK_[REGLA]_t[###][tipo]_[nombre_campo]
NE  ejemplo : CONSTRAINT `CHK_NE_t125t_code`   CHECK (`code`   <> _utf8mb4'')
GT0 ejemplo : CONSTRAINT `CHK_GT0_t520m_price`   CHECK (`price`    > 0)
LEN ejemplo : CONSTRAINT `CHK_LEN_t125t_iso_639` CHECK (LENGTH(`iso_639`) = 4)

-- REGLAS GENERALES --
- Siempre en mayusculas el prefijo y la regla
- El nombre de la tabla siempre en minusculas
- El nombre del campo siempre en minusculas
- Sin espacios ni caracteres especiales
